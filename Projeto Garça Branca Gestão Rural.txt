Projeto Garça Branca Gestão Rural
Versão: 1.0 Status: Documento-base para desenvolvimento inicial Responsável pelo projeto: Gabriel Perin Usuários previstos: 3 usuários principais — Gabriel, pai e mãe Interface principal desejada: WhatsApp Banco de dados principal: Postgres via Supabase Objetivo: construir um sistema próprio, econômico, seguro e personalizado para gestão da fazenda, gado, funcionários, financeiro, estoque, documentos, manejo, cascalheira e supressão.

1. Visão geral do projeto
O projeto Garça Branca Gestão Rural será um sistema familiar de gestão rural criado especificamente para a rotina da fazenda dos pais de Gabriel.
A fazenda/sede principal é conhecida como Garça Branca, localizada na região de Extrema/RO, aproximadamente no km 1010. A operação envolve criação e venda de gado para frigorífico, controle de lotes, pastos, funcionários, compras, despesas, documentos, manejo, cascalheira, supressão e serviços de campo.
O sistema deve funcionar principalmente pelo WhatsApp, para que os usuários não precisem aprender a usar um app complexo. A ideia é que os usuários mandem mensagens, áudios ou fotos, e o sistema transforme essas informações em registros organizados.
Exemplo:
Usuário envia no WhatsApp:
Comprei 10 sacos de sal por 1.800 no Pix.
O sistema interpreta, pede confirmação e salva como:
	•	despesa;
	•	entrada de estoque;
	•	item: sal mineral;
	•	quantidade: 10 sacos;
	•	valor: R$ 1.800;
	•	forma de pagamento: Pix;
	•	data;
	•	usuário responsável;
	•	anexo, se houver.
O sistema também deve permitir consultas pelo WhatsApp, como:
	•	“Quanto gastamos esse mês?”
	•	“Quantas cabeças temos hoje?”
	•	“Quanto gastamos com sal?”
	•	“Quanto João recebeu de adiantamento?”
	•	“Qual venda ainda não foi recebida?”
	•	“Tem tarefa pendente?”
	•	“Tem conta vencendo?”
	•	“Quando foi a última pesagem?”
	•	“Quanto cascalho já foi retirado?”
	•	“Tem licença ou autorização vencendo?”

2. Objetivo principal
Criar um sistema que transforme a rotina rural em dados confiáveis.
Atualmente, muitas informações podem estar espalhadas entre:
	•	memória;
	•	caderno;
	•	conversas de WhatsApp;
	•	fotos soltas;
	•	comprovantes;
	•	recibos;
	•	notas;
	•	áudios;
	•	mensagens de funcionários;
	•	documentos físicos;
	•	informações lembradas por pai/mãe.
O sistema deve centralizar tudo em uma base organizada e segura.
Objetivo final:
Tudo que acontece na fazenda deve virar registro: gado, dinheiro, funcionário, estoque, venda, pasto, documento, manejo, cascalheira, supressão e tarefa.

3. Princípios do sistema
3.1 Simplicidade para os usuários
Os usuários principais não devem precisar abrir um sistema complicado.
A experiência ideal é:
	•	Usuário manda mensagem, áudio ou foto no WhatsApp.
	•	Sistema entende.
	•	Sistema pede confirmação quando necessário.
	•	Sistema salva no banco de dados.
	•	Usuário consulta pelo WhatsApp quando quiser.
O WhatsApp será a interface principal.
3.2 Segurança dos dados
Os dados da fazenda são importantes. O sistema não pode depender apenas de planilha.
A planilha pode ser usada para:
	•	exportação;
	•	backup adicional;
	•	conferência;
	•	envio ao contador;
	•	relatório externo.
Mas a base principal deve ser um banco de dados confiável.
Banco principal recomendado:
Supabase/Postgres
3.3 IA não é a fonte da verdade
A IA deve apenas interpretar mensagens, áudios e fotos.
A verdade do sistema é o banco de dados.
A IA não deve inventar dados nem responder consultas “de memória”. Para responder perguntas, ela deve consultar os registros salvos.
3.4 Confirmação antes de ações críticas
Ações importantes não devem ser salvas automaticamente sem confirmação.
Ações críticas:
	•	venda de gado;
	•	morte de animal;
	•	nascimento;
	•	alteração na quantidade de cabeças;
	•	movimentação de lote;
	•	pagamento de funcionário;
	•	adiantamento;
	•	baixa de estoque;
	•	registro de supressão;
	•	retirada de cascalho;
	•	documento ambiental;
	•	alteração de venda;
	•	exclusão/cancelamento de registro.
Sempre que a IA interpretar uma ação crítica, o sistema deve responder:
Entendi isso: […]. Confirmar?
Somente após confirmação o registro deve ser oficialmente salvo.
3.5 Nada deve ser apagado definitivamente
Registros importantes nunca devem ser deletados de forma permanente.
Se houver erro, o registro deve ser:
	•	cancelado;
	•	corrigido;
	•	substituído;
	•	marcado como duplicado;
	•	arquivado.
O histórico deve permanecer.
3.6 Tudo deve ter autoria e data
Todo registro precisa guardar:
	•	quem criou;
	•	quando criou;
	•	origem do registro: WhatsApp, painel, importação etc.;
	•	mensagem original;
	•	anexo original, se houver;
	•	quem confirmou;
	•	quando confirmou;
	•	alterações posteriores.

4. Escopo inicial do sistema
A primeira versão não deve tentar fazer tudo de uma vez.
O sistema deve ser construído em fases.
Fase 1 — Fundação
Objetivo: criar banco de dados, painel simples e registros manuais.
Incluir:
	•	usuários;
	•	fazendas;
	•	pastos;
	•	lotes de gado;
	•	funcionários;
	•	despesas;
	•	receitas;
	•	vendas;
	•	estoque;
	•	tarefas;
	•	documentos;
	•	auditoria.
Não implementar WhatsApp ainda.
Fase 2 — IA de texto
Objetivo: interpretar mensagens escritas em português.
Exemplos:
	•	“João recebeu 300 de adiantamento”
	•	“Comprei 10 sacos de sal por 1.800”
	•	“Nasceram 3 bezerros no pasto 4”
	•	“Vendi 18 bois para o frigorífico”
	•	“Tiramos 4 cargas de cascalho”
	•	“Cerca do pasto 3 caiu”
A IA deve transformar a frase em JSON estruturado.
Fase 3 — WhatsApp com texto
Objetivo: conectar WhatsApp Cloud API.
Fluxo:
	•	WhatsApp recebe mensagem.
	•	Webhook recebe a mensagem.
	•	Sistema salva mensagem bruta.
	•	IA interpreta.
	•	Sistema cria ação pendente.
	•	Usuário confirma.
	•	Sistema salva.
	•	Bot responde.
Fase 4 — Áudio
Objetivo: receber áudio pelo WhatsApp.
Fluxo:
	•	Usuário envia áudio.
	•	Sistema baixa o áudio.
	•	Sistema salva o arquivo.
	•	IA transcreve.
	•	IA interpreta.
	•	Sistema pede confirmação.
	•	Sistema salva.
Fase 5 — Foto de nota e comprovante
Objetivo: receber imagem pelo WhatsApp.
Fluxo:
	•	Usuário envia foto.
	•	Sistema salva imagem.
	•	IA tenta extrair dados.
	•	Sistema identifica valor, data, fornecedor, item e tipo.
	•	Sistema pergunta se deve registrar.
	•	Usuário confirma.
	•	Sistema salva e anexa a imagem ao registro.
Fase 6 — Consultas pelo WhatsApp
Objetivo: responder perguntas com base no banco de dados.
Exemplos:
	•	“Quanto gastamos esse mês?”
	•	“Quanto gastamos com sal?”
	•	“Quantas cabeças temos?”
	•	“Qual lote está no pasto 3?”
	•	“Quanto João recebeu esse mês?”
	•	“Quais vendas estão pendentes?”
	•	“Tem conta vencendo?”
	•	“Tem tarefa atrasada?”
	•	“Quanto cascalho já tiramos?”
Fase 7 — Alertas automáticos
Objetivo: o sistema avisar coisas importantes.
Alertas:
	•	conta perto de vencer;
	•	venda ainda não recebida;
	•	funcionário para pagar;
	•	adiantamento pendente;
	•	estoque baixo;
	•	tarefa atrasada;
	•	pesagem atrasada;
	•	manejo pendente;
	•	documento/licença vencendo;
	•	comprovante faltando;
	•	autorização de supressão vencendo;
	•	retirada de cascalho sem documento anexado.
Fase 8 — Painel melhorado
Objetivo: painel web para Gabriel revisar, corrigir e administrar.
Funções:
	•	ver registros;
	•	filtrar por módulo;
	•	corrigir lançamentos;
	•	cancelar registros;
	•	ver anexos;
	•	exportar planilhas;
	•	ver alertas;
	•	cadastrar dados estruturais;
	•	revisar ações pendentes.

Inicialmente:

Objetivo do backend:

	•	chamar IA;
	•	salvar no banco;
	•	buscar dados;
	•	enviar respostas;
	•	rodar alertas.
5.4 Banco de dados
Principal:
	•	Supabase/Postgres
Motivo:
	•	banco relacional confiável;
	•	suporta tabelas complexas;
	•	permite auditoria;
	•	permite Row Level Security;
	•	integra autenticação;
	•	integra storage;
	•	permite APIs automáticas;
	•	tem plano gratuito para testes;
	•	pode migrar para plano pago para uso real.
5.5 Storage
Para guardar:
	•	fotos de notas;
	•	comprovantes Pix;
	•	recibos;
	•	documentos;
	•	imagens de campo;
	•	fotos de pasto;
	•	fotos de cascalheira;
	•	fotos de supressão;
	•	áudios originais.
Arquivos não devem ser salvos diretamente como blobs no banco principal.
O banco deve guardar:
	•	link do arquivo;
	•	tipo;
	•	tamanho;
	•	usuário;
	•	data;
	•	módulo relacionado;
	•	registro relacionado.
5.6 WhatsApp
Usar:
	•	WhatsApp Cloud API oficial da Meta.
Funções:
	•	receber texto;
	•	receber áudio;
	•	receber imagem;
	•	enviar mensagens;
	•	enviar alertas;
	•	pedir confirmação;
	•	responder consultas.
5.7 IA
Usar uma camada abstrata chamada AIProvider.
Inicialmente pode ser:
	•	OpenAI API;
	•	Gemini API.
O código deve ser preparado para trocar o provedor depois sem reescrever o sistema inteiro.
A IA será usada para:
	•	interpretar texto;
	•	transcrever áudio;
	•	ler imagem;
	•	classificar mensagens;
	•	extrair campos;
	•	sugerir ação;
	•	responder em linguagem natural com base em dados consultados.
A IA não deve fazer alterações diretas no banco sem passar pela camada de validação.

12. Segurança e confiabilidade
Regras obrigatórias
	•	Nenhuma tabela crítica deve deletar dados definitivamente.
	•	Toda ação crítica exige confirmação.
	•	Todo registro deve guardar o usuário responsável.
	•	Toda mensagem bruta deve ser salva.
	•	Todo anexo deve ser salvo.
	•	Toda alteração deve gerar log.
	•	Deve existir exportação de dados.
	•	Deve existir rotina de backup.
	•	Deve existir controle de duplicidade.
	•	A IA não pode executar alteração sem validação.
Controle de duplicidade
O WhatsApp envia identificador único de mensagem.
Salvar como:
	•	external_message_id
Criar constraint unique nessa coluna.
Se o mesmo webhook for recebido duas vezes, o sistema deve ignorar a duplicata.

Não construir primeiro um app bonito.
